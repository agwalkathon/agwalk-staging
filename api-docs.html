Overview

The Arcgate Walkathon application integrates directly with the Strava API to track and reward walking and running activities for employees. This document is a centralized reference outlining how data flows from Strava, how it is processed by the backend scoring engine, and how it is saved and summarized in Supabase.

Key Architecture

The system is split into three main components:
1. Backend Server (Node.js/Express): Handles Webhooks, token refreshes, sync triggers, and runs the scoring engine.
2. Supabase (PostgreSQL): Holds activities, challenge configurations, athlete settings, and points summaries.
3. Frontend Portal (HTML/JS): Renders leaderboard statistics, user dashboards, and organizers' administrative controls.

Participant Registration & Approval Flow

The onboarding pipeline secures participant entries by verifying profiles against an active employee allowlist before authorizing Strava connectivity.

Onboarding Sequence Overview

Step 1: Registration Request (new-participant.html) ➜ Step 2: Employee Verification (employees_master lookup) ➜ Step 3: Pending Entry Creation (registration table) ➜ Step 4: Strava OAuth Redirect (connect-strava.html) ➜ Step 5: Authorization Token Exchange (POST /register) ➜ Step 6: Organizer Approval (athletedtl.html).

Detailed Onboarding Lifecycle

  1. Initial Registration Form Submission (POST /register-request):
    An athlete fills in details on the registration form. The backend verifies the email and employee code against the employees_master table to ensure the user is an active employee.
    If verified, an entry is created in the registration table with status = 'pending', role = 'user', and strava_connected = false.
  2. OAuth Handshake with Strava:
    Once the employee registration entry is saved, the user is redirected to Strava to authenticate. Upon consent, Strava redirects back to connect-strava.html with an authorization code.
  3. Token Exchange and Connection (POST /register):
    The frontend sends the authorization code to the backend. The backend calls the Strava OAuth endpoint (https://www.strava.com/oauth/token) to swap it for a refresh_token, access_token, and the athlete profile.
    The backend updates the registration entry with:
    • strava_athlete_id (Strava athlete profile ID)
    • access_token & refresh_token (encrypted tokens)
    • profile_photo (URL to Strava profile image)
    • strava_connected = true
    • strava_connected_at = NOW()
  4. Organizer Review & Approval (athletedtl.html):
    Organizers inspect the pending queue. Approving a user triggers a database PATCH setting status = 'approved'. This activates the user on the leaderboard and schedules history synchronization.

Scoring Rules & Engine

Points are calculated per athlete per calendar day using their local activity_date_time_ist date.

1. Base Points

  • Calculated as 1 Point per 1 km walked.
  • Distances are not rounded down; e.g. walking 5.45 km grants 5.45 base points.

2. Milestone Bonuses (Base Tier)

Athletes earn extra points based on their cumulative distance on any single calendar day:

Daily Distance Milestone Bonus Points Awarded Total Earned (Base + Bonus)
5.0 km to 7.99 km +1.0 Point Distance + 1.0
8.0 km to 9.99 km +2.0 Points Distance + 2.0
10.0 km to 14.99 km +3.0 Points Distance + 3.0
15.0 km to 20.99 km +4.0 Points Distance + 4.0
21.0 km or more +5.0 Points Distance + 5.0 (Capped at 26.0)

3. Daily Cap (21 km / 26 Points Limit)

To ensure fair competition, active walking distance is capped at 21 km per calendar day (maximum 26.0 total points including the 5.0 milestone bonus).

Special Double Point Days

Certain dates loaded into the special_days table bypass the cap or multiply scoring. On these dates, the caps are adjusted programmatically (e.g. 2x points or a higher cap).

4. Range-Split Active Challenges Logic

If a challenge on a day is a range-split type (e.g. walk 5km for a challenge):

  1. We identify the activities matching the challenge.
  2. We deduct the challenge distance (e.g. 5km) from the day's total walking pool.
  3. The deducted distance is scored separately as Challenge Points.
  4. The remaining distance is evaluated against the milestone tiers to compute Base Points + Milestone Bonuses.

5. Multi-Challenge Days

If an athlete enters two challenges on a single day, the engine resolves them in order of priority, separating challenge pools to prevent double-dipping, and scores the remaining remainder towards daily base standings.

Steps Resolution Rules

Because some devices do not write steps directly to Strava, the steps count is resolved via the following prioritization pipeline in resolveSteps(a):

function resolveSteps(activity) {
  // Tier 1: Strava native steps field
  if (activity.steps && activity.steps > 0) {
    return activity.steps;
  }
  // Tier 2: Stride cadence calculation
  if (activity.average_cadence && activity.moving_time) {
    return Math.round(activity.average_cadence * 2 * (activity.moving_time / 60));
  }
  // Tier 3: Distance estimates based on sport type
  const distKm = (activity.distance || 0) / 1000;
  if (activity.sport_type === 'Walk' || activity.sport_type === 'Hike') {
    return Math.round(distKm * 1350); // 1,350 steps/km
  }
  if (activity.sport_type === 'Run' || activity.sport_type === 'VirtualRun') {
    return Math.round(distKm * 1150); // 1,150 steps/km
  }
  return null;
}

Social Feed & Celebrate Settings

The social feed (announcements) and the celebrate feed push achievements to participants in real time. Configuration is managed in feed-config.html and stored in the leaderboard_config table.

1. Feed Configuration JSON Schema

The configuration row matches config_key = 'feed_config' in `leaderboard_config` and stores the following JSON payload:

{
  "rules": {
    "allow_standard_activities": true,  // Display standard daily walks
    "allow_medals": true,               // Celebrate milestones (clubs, streaks)
    "allow_distance_clubs": true,       // Highlight 100k, 200k, 300k clubs
    "allow_rank_top1": true,            // Push rank #1 shifts
    "allow_flagged_activities": false,  // Block flagged walks from appearing
    "allow_private_profiles": true      // Allow private users' details (obfuscated)
  },
  "filters": {
    "minimum_activity_distance_km": 1.0,
    "allowed_sports": ["Walk", "Run", "Hike", "Ride"]
  }
}

2. Announcement Pipeline (ensureActivityAnnouncement)

When an activity is synced, it passes through the filter check:

  • Checks if a duplicate announcement exists for title = 'Activity:{strava_activity_id}'.
  • Enforces participant exclusion settings: Admins, system roles, flagged athletes, and private profile toggles.
  • Filters out activities that fall below minimum_activity_distance_km or sport types not included in allowed_sports.
  • Inserts a record into the announcements table with the body stringified as JSON.

3. Celebrate Page configuration (celebrate-config.html)

Controls event celebration styling (e.g. background effects, sound highlights, visual layouts, and color themes). Settings are saved under config_key = 'celebrate_config' in `leaderboard_config`.

Push Notification Engine

Push notifications allow organizers to alert users of progress milestone achievements (clubs, streaks, long walks, and daily updates) using W3C WebPush subscriptions.

1. WebPush Subscriptions Flow

  1. Subscription Registration (POST /push/subscribe):
    The frontend portal requests push credentials from the browser. The resulting subscription object is posted to the backend along with the athlete_id and device_name.
    The backend inserts/patches the subscription JSON directly into the push_subscriptions table.
  2. Removing Inactive Keys:
    If the WebPush dispatch (via webpush.sendNotification) fails with status 410 (Gone), the backend automatically deletes the inactive subscription record.

2. Trigger Keys and Default Templates

Rules are stored in `push_notification_rules`. If a rule lacks a custom override, the system falls back to default notification templates:

Database Schema & Fields

Below are the primary Supabase tables used for tracking and scoring activities.

1. registration (Athlete Details)

Stores the registered user profiles, shift details, and OAuth tokens.

  • strava_athlete_id: Unique numeric id returned by Strava.
  • full_name: Full name of the participant.
  • emp_code: Unique employee code.
  • shift: Athlete's work shift (e.g. Dayshift, Nightshift, etc.).
  • access_token & refresh_token: API credentials used to pull data.
  • strava_connected: Boolean flag showing active integration.

2. activities (Raw Activities)

Stores individual activity logs fetched from Strava API.

  • activity_date_time_ist: Stores the local calendar date and time. **The primary field used for daily groupings on leaderboards.**
  • activity_date: The raw UTC time stored for safety.
  • start_time: Time portion parsed from the local date (e.g., 05:03:42).
  • distance_meters: Float distance.
  • moving_time_seconds: Float time duration.
  • is_flagged: Indicates if the activity violates validation rules (e.g., pace too fast, overlapping times).

3. athlete_points_summary (Cached Standings)

Holds pre-calculated aggregated statistics for high-performance frontend loads.

  • total_distance_km: Cumulative distance across the event.
  • base_points: Base score awarded from active walking.
  • challenge_points: Points earned from participating in specific challenges.
  • bonus_points: Cumulative milestone/special day bonuses.
  • total_points: base_points + challenge_points + bonus_points. **This cached value is queried by leaderboards.**

4. challenges (Challenge Targets)

Maintains dynamic active challenges.

  • distance_type: Can be fixed (e.g., complete exactly 5km) or range (split-based logic).
  • distance: The threshold target in km.
  • points: The points rewarded if achieved.

API Endpoints

The backend server hosts the following crucial endpoints for syncing and administration:

Sync Operations

  • POST /admin/refresh-strava-tokens: Iterates through registered athletes, checks oauth expiry, and retrieves fresh access tokens from Strava.
  • POST /admin/backfill-points: Recalculates and caches points summaries for all registered participants. Runs when adjustments to milestones or past errors are made.
  • POST /register: Webhook endpoint called by Strava's servers during athlete authorization.

System Settings

  • GET /maintenance-status: Light API checking if the portal is in maintenance mode. Allows bypassing for administrators.
  • PATCH /admin/maintenance: Admin endpoint to toggle maintenance mode, add whitelist IDs, and write system announcement text.

Common Fixes & Playbook

1. Recalculating Leaderboard Points

If points are showing incorrectly for a participant due to a configuration change or timezone update, run a full recalculation of points summaries:
Navigate to the Data Upload page and hit the Recalculate Summary button. This updates the cached table in Supabase.

2. Clear Cache Button

If frontend points are desynced from raw database values, click the Clear App Cache button in the admin banner. This clears cookies, sessionStorage, and deletes browser service-worker cache keys forcing a hard refetch of the stats API.

3. Repairing a Single Incorrect Activity Date

If an activity is placed under the wrong date (e.g. UTC offset desync), you can manually update it in Supabase: 

UPDATE activities 
SET activity_date_time_ist = '2026-06-25T05:03:42' 
WHERE strava_activity_id = 19055574519;
After applying the change, run the Recalculate Summary operation to sync the leaderboard totals.
